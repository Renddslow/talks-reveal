In those days, web sites were individual files hosted on server that could respond to HTTP requests. It made these requests one at a time. So once the client or browser had received the HTML file for the page, it might have to make a return trip to the server to get a stylesheet for the page.

If you were really fancy, you'd include JavaScript. Mostly for pointless things like visitor counts and annoying pop ups.

All of this content had to be downloaded in order before the browser could render any of your page.

All of this content had to be downloaded in order before the browser could render any of your content.

So let's walk through what a typical page load would look like for Pinterest in 1999.

Pinterest.com in 1999

  • Browser: "Hey server!"
  • Server: "Hi."
  • Browser: "I see you're awake."
  • Browser: "I'd like index.html, please."
  • Server: ...
  • Server: ...
  • Server: "Found it, here you go!"
  • Browser: "Oh boy! Thanks!"

Now once its had the conversation and gotten index.html, it's going to scan through that to see if there's some JavaScript files or Stylesheets. Then one by one its going to have the exact same conversation above until it has everything and can render the page. Now once it has all of those, it will render the page and start having the exact same conversation again for every image on the page. And if you've ever used Pinterest, that's a lot of images.

Midway through the 90's and into the early 00's Flash web sites and games became the new hotness.
I mean, look at this cutting edge 3D rendering. You couldn't do this without Flash.
YouTube was launched by former PayPal engineers using the power of Flash to play compressed videos. Suddenly your pages could be incredibly interactive.

Flash sites disadvantaged users by requiring additional software running on their machines and disadvantaged assistive technology users.

But it came at a cost. Your users had to have the right version of Flash installed on their local machines. It also made it difficult for folks with disabilities using assistive technology to interact with the pages as everything was under the Flash hood. By 2007, Flash was the dominate force in web applications.
But in 2007, everything changed...

Apple Released the iPhone

But everything changed in 2007 with the release of the iPhone and the App Store. Apple made a conscious decision to not support the Flash player on their hardware, meaning that a major portion of the web wasn't accesible from the iPhone.
  • Asynchronous
  • JavaScript
  • And
  • XML
Now just two years prior 2005, Google started utilzing a technology bundled in most browsers since 98 to make calls to their server without having to reload the page. This pattern and associated technologies became known as Asynchronous Javascript and XML.

AJAX

All of a sudden, you didn't necessarily need to reload your page to get updates from the server.

This led to a rise in applications that co-opted the power of a growing number of server-side frameworks like Ruby on Rails, Django, or ASP.NET which would supply the initial data and templating to a page, and then with the power of AJAX, we could check for updates, and update CRUD forms without a reload.

HTML5 gave the browser more power, reducing the need for the server to generate markup.

With the rollout of HTML5 between 2011 and 2014 the need for server-side templating started to evaporate. The browser suddenly had power to maintain its own navigation state without reaching out to a server. In fact, the client could be completely decoupled from a server and still have the same, or better, experience.

Web Apps Today

This increasingly common form of web applications is called a Single Page App.
Before we dig into some of the technical features of single page apps, I want to talk about what value SPA's provide to their users.

What the user gets

  • Single page
  • Browser-side caching
  • Faster navigation between screens
  • Perceived performance increase during loading states

What the developer gets

The key to all of this is JavaScript

HTML5 gives us better interaction with the DOM

Massive improvements over the past 20 years both to the language itself and the wider JavaScript ecosystem mean we get to do really cool stuff right in the browser all natively. Things like data-visualization, 3D-rendering, video games, and live video chat.

With the rollout of HTML5 we also gained much better access to the browser's way of understanding and modifying the HTML documents called the DOM.

What the developer gets

  • History
  • Browser Storage
  • Fetch

We also gained features like History, Browser Storage, and Fetch.

History is the feature that allows us to handle navigation all from the browser. It basically tricks your browser into thinking it's changed pages despite the fact that its still on a simple index.html page. JavaScript is then able to intercept these History changes and respond with new content.

Browser storage allows us to safely* store user or application data scoped directly to your domain. This is a massive improvement over cookies which, while technically can be used to store information by the browser, are passed in the header of every HTTP request made to a server. In many cases, these Cookies can be picked up by other sites or intercepted by bad actors. That's not to say that Cookies don't have their place, but Storage gives us a safer place to keep things that don't need to be transferred over every request.

Finally the Fetch API allowed us to make AJAX requests in a much simpler way while leveraging some of the improvements in JavaScript's asynchronous interface.

All of these features are used in tandem to create powerful application experiences that in the past were only possible directly on the native desktop or through third-party tools like Flash.

Let's Talk About JavaScript

There are three common critiques that tend to come up from non-frontend developers when talking about the JavaScript ecosystem.

"JavaScript is the worst! Why do we have to use it!? Why is nothing typed strongly and everything dynamic????"

  • JavaScript interfaces with the DOM.
  • JavaScript is event driven.
  • JavaScript is the language of the web...
  • ...and it's all we've got...
  • ...but it's constantly getting better.

The first one is often, "JavaScript is absolutely the worst. Why do we have to use it? It can't even do loose typing correctly!"

JavaScript gives us an interface with interacting with the document on the user's screen in an easy to use manner. This isn't your father's DOM. JavaScript is largely event-driven, meaning we can respond to user's interaction with the screen interactively, but only respond to the actions we care about.

"Why do I need a framework?"

  • Frameworks are designed to help developers think.
  • Frameworks help us focus on big important problems and abstract away repeated browser things.
  • Frameworks each solve different problems well.
  • Focus on solving big problems and not on frameworks.

Okay so I'm not going to answer the jquery question, but if you are curious I would be happy to talk your ear off about it for like 5 minutes after this.

So why do we have frameworks? It's super simple. Frameworks are designed to help developers think. They give us a shared set of patterns that mean we don't have to reinvent the wheel every time we approach a tough problem. Each framework is designed with a different usecase in mind. For Installed Sales, we've adopted React. Some of the things React is good at doing for us is managing application state, determining when we need to render or rerender a screen, and ensure that data is always going only one direction. Now for Installed Sales, these were exactly the tools we needed. For some teams in the broader ecosystem, you might need a more reactive rendering environment for data visualization, so you pick up Svelte. Or perhaps you need two-way data flow, so you pick up Angular. These are simply tools that let us developers focus on big problems, while abstracting away the common and oft repeated ones.

"It takes so dang long to set up my frontend dev environment!"

  • Yes.
  • We're working on it.
  • It's getting better.
The last thing that often comes up, its that final swing people try to take at the frontend ecosystem is that it just takes so dang long to set up your local development environment. There's all those build steps and dependencies, and have you checked out the size of your node_modules folder recently!? All I have to say about that is, yes, you're 100% right. If you're dipping your toe into the frontend waters for the first time your initial set up will be a little bit painful, but what dev environment isn't? The big value of your node_modules folder is that it doesn't have to download dependencies at every runtime and takes care of a lot of the problems of interdependent dependencies. And this is getting better with every major release of the module systems in both Node and npm.
